# Ubuntu

### 1.三种网络连接方式(针对虚拟机)

#### 1.1.桥接模式:

*   系统作为一个**独立**的网络IP，直接从路由器中获取网络IP的地址，可以直接上网，同时外部设备也可以直接访问此IP地址

###### <u>适用场景</u>：

*   希望虚拟机里面的linux系统可以像主机一样加入到网络里面，同时需要在局域网里面测试服务器，比如搭建一个网站能够让其他设备访问

#### 1.2.NAT模式：

*   NAT模式则是让系统隐藏在你的主机系统后面，通过主机来和外部进行网络沟通，不会产生独立的IP地址，通过主机的IP地址**借道上网**，相对比较安全

###### <u>适用场景</u>：

*   使用虚拟机来上网，比如学习或者测试软件
*   或者说不想暴露在互联网中，注重安全型
    ***注意***：由于没有独立的IP，无法单独搭建网站来使得其他用户来进行访问

#### 1.3.主机模式：

*   主机模式即主机和linux系统搭建的私人通信模式，完全隔断互联网，不能上网，通过一个私有IP实现主机和linux系统的互相访问，但是不能访问外部网站或服务

###### <u>适用场景</u>:

*   虚拟机用来搭建一个本地服务器，然后用物理上的主机来测试，仅仅是主机有访问权限用来测试，即作为一个安全隔离的实验的环境

### 2.网络远程连接:

#### 2.1获取ip地址:
- **`ip addr show`** : 该命令可以获取
    - > 1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue **state UNKNOWN** group default qlen 1000
      link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
      inet 127.0.0.1/8 scope host lo
       valid_lft forever preferred_lft forever
      inet6 ::1/128 scope host 
       valid_lft forever preferred_lft forever
      2: ens33: <BROADCAST,MULTICAST> mtu 1500 qdisc noop **state DOWN** group default qlen 1000
      link/ether 00:0c:29:1d:8e:4a brd ff:ff:ff:ff:ff:ff
        - >**命令返回值解析**：
            >**`lo`**: 回环接口，用于测试 本地通信的网络栈问题，这里的    
            > **`<LOOPBACK,UP,LOWER_UP\>`** 显示接口启用
            >**`LOWER_UP`** 代表底层链路正常工作
            >**`state unknown`** 代表无影响。
            >**`MTU 65536`** 表示最大传输单元为65536字节。
            >**`inet`** 代表IPV4,本地回环地址(scope host)
            >**`inet6`** 代表IPV6，也是本地回环地址
            >**`ens33`** 代表以太网接口(这里的en则代表网络接口.s33为编号)用于连接网络
            >**`link/ether 00:0c:29:1d:8e:4a`** 是接口的硬件地址(MAC地址)
            >**`brd ff:ff:ff:ff:ff:ff`** 是广播地址。
            >硬件地址用来作为以太网的寻址，确保正确网络传输，广播地址则用于向局域网内所有设备发送数据的特殊地址。
            >**<u>综上所述</u>:** **这里显示本地网络正常，但是以太网接口没有启用，需要手动启用**
- **`sudo ip link set ens33 up`** : 启用以太网接口
- **`sudo dhclient ens33`** ： 手动触发 DHCP 请求以获取 IPv4 地址
    - 这里有个重要的点，如果使用DHCP触发，那么就无法托管到Networkmanager上面 
- 然后调用 **`ip addr show`** ：
    - >1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN group default qlen 1000
      link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00

       valid_lft forever preferred_lft forever 
        - >这里的 **`state up`** 代表以太网接口已启用，且分配了IPV4和IPV6接口

#### 2.3 管理自动启用和获取IP地址进行网络连接
- 每个系统运行的主机都有自己的以太网接口名称（基于MAC地址），在目前的虚拟机上，名称为 **<u>ens33</u>**
- 运行  **`nmcli connection show`** 后没有任何网络连接被列出。这表明当前系统中没有通过 **NetworkManager**  配置的网络连接，可以使用networkmanager 来管理自动获取网络连接
    -  **`nmcli connection show`**，查看当前网络连接(前提是采用NetworkManager来管理网络，手动启用则不会显示)

- **`sudo nmcli connection add type ethernet ifname ens33 con-name "Wired Connection"`** 
    - **`type ethernet`**：指定连接类型为以太网。
    - **`ifname ens33`**：指定接口名称为 ens33,根据你的MAC地址可以更改
    - **`con-name "Wired Connection"`**：为连接命名（可以自定义）

- **`sudo nmcli connection modify "Wired Connection" ipv4.method auto`** 
    - 设置 DHCP 自动获取 IP，这里是ipv4

- **`sudo nmcli connection modify "Wired Connection" connection.autoconnect yes`** 
    - 确保连接在系统启动时自动启用
- **`sudo nmcli device set ens33 managed yes`**  强制对响应的接口进行托管

- **`sudo nmcli connection up "Wired Connection"`** 
    - 激活新创建的连接

- **`nmcli device status`**
    - 验证是否托管 

- **`sudo nmcli connection delete "Wired Connection"`**
    - 如果需要删除该网络则调用这个命令 

##### 2.3.1解决无法托管的问题步骤
>由于更改权限有点麻烦，需要去配置PolicyKit权限限制，这里都是基于root直接越过权限限制
- **`nmcli device status`**: 检查是否托管

- **`systemctl status NetworkManager`**: 查看当前是否启用 <u>`NetworkManage`</u>  工具

- 首先如果不小心启用了DHCP(**`sudo dhclient ens33`**)手动开启了网络请求，此时是无法采用NetworkManager来进行托管的，此时使用 **`ps aux | grep dhclient`** 命令去找出当前的 dhclient 进程   
    - 然后调用 **`sudo kill 2294`** 来杀死当前的DH进程，这里的2294取决于前面查找出来的dhclient进程的端口号
    - 然后调用 **`sudo dhclient -r ens33`** 来对**ens33**接口的DHCP解除租约，不再通过DHCP去获取网络
    - 然后可以通过 **`sudo nano /etc/NetworkManager/NetworkManager.conf`** 进入到**NetworkManager**的文件配置界面
        - 里面确保有以下内容： 
            >[ifupdown]
            >managed=true
            
            >[device]
            >match-device=interface-name:ens33 (注释：ens33 可以改为其他接口)
            >managed=true
        - 这里的 sudo 代表root权限，越过policykit，直接编辑
        - nano 为编辑工具(ctrl + x为退出)
    - 配置完**networkmanager**后需要使用 **`sudo systemctl restart NetworkManager`**命令来重启
    - 重启后需要清理遗留的状态再继续开启
        - **`sudo systemctl stop NetworkManager`** : 停止当前的networkmanager任务
        - **`sudo rm -f /var/lib/NetworkManager/NetworkManager.state`**: 清理缓存，防止状态未更新，无法完成托管
        - **`sudo systemctl start NetworkManager`**：开启networkmanager托管网络
    - 此时直接强制对响应的接口进行托管，**`sudo nmcli device set ens33 managed yes`**

### 3.终端(shell)的使用方式：

#### 3.1.相关特性：
* **`sudo -i`**： 进入root模式，代表超级用户权限

*   终端的使用是基于当前文件夹目录的，即在一个文件夹打开终端，那么终端的工作目录即为当前文件夹，作为<u>**默认工作路径**</u>，例如：

    *   **`ls`**： 会列出当前工作目录中的文件和子文件夹
    *   **`pwd`**: 查看当前的工作目录，类似于/home/user/documents
        * **`pwd`** = print working directory

*   前面指明了默认工作路径，那么当然也可以指定其他工作路径,例如：

    *   **`cd /path/to/directory`**: cd命令是进入指定的文件夹为工作目录
    *   **`cat filter.txt`**：查找指定的文件，且将文件内容输出到窗口里面

#### 3.2.终端的环境变量(path)

*   打开终端会默认加载一个默认的path，包含了系统中一些常见的目录路径，用于查找可执行文件，如果想要查看，可以使用`echo $PATH`,以冒号（:）分隔

    *   **`echo $PATH`**: 获取当前的PATH，弹出 **`/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/usr/games:/usr/local/games:/snap/bin`**

###### 返回的相关命令含义:

*
    *   **`/bin`**：存放基本命令（如 ls、cp、mv 等），对所有用户可用。
        *   **`/usr/bin`**：存放更多用户级命令（如 python、gcc 等）。
            *   **`/usr/local/bin`**：存放用户安装的本地程序。
                *   **`/sbin 和 /usr/sbin`**：存放系统管理员使用的命令（如 shutdown、ifconfig 等）。
                    *   **`/usr/local/sbin`**：存放用户安装的系统管理工具。
                        *   **`/snap/bin`**: 存放通过Sanp包管理器安装的应用程序的可执行文件

##### 3.2.1.Path实际使用方式

###### 执行可执行文件，采用直接执行的方式:

*   想要执行当前工作目录里面的可执行文件，一般通过相对路径驱动，例如：

    *   **`./hello`**:  "./"代表相对路径，此命令代表执行名称为hello的可执行文件，通常不需要后缀名

###### 添加可执行文件放入PATH里面，分为临时和永久:



- - ###### **临时追加**:

    *   **`export PATH=$PATH:.`**: **'.'** 代表当前目录，通过这行命令临时将该文件添加到PATH里面



- - ###### **永久追加**:

    *   **`echo 'export PATH=$PATH:.' >> ~/.bashrc`**: 将当前目录追加到PATH中，并且打印出来
    *   **`echo`**：打印字符串
    *   **`>>`**: 作为操作符，将\*\*`echo`\*\* 的输出追加到文件末尾
    *   **`export PATH=$PATH:.`**:这也是临时追加到PATH里面的命令，在这里用来作为永久追加到PATH中的命令行中
    *   **`~/.bashrc`**: <u>这是 Bash 的用户级配置文件，每次启动非登录 Shell 时会自动加载它。</u>
    *   **`source ~/.bashrc`**: 用于重新加载指定的配置文件，而无需重启客户端了

### 4.Linux目录结构

>**相比于windows的C盘，Linux加载的目录更多更杂，整体挂在一个目录数上，在结构体上都是基于根目录'/'的基础上创建其他目录，整体采用树状分布**


#### 4.1 `/dev` - 设备文件

*   **用途**: 存放设备文件，包含硬件设备的特殊文件（如磁盘、终端）。
*   **示例**: 存储块设备和字符设备文件。
*   **备注**: 如 `/dev/sda` 表示第一个磁盘。

#### 4.2 `/media` - 可移除媒体

*   **用途**: 用于挂载可移除媒体（如 U 盘、CD-ROM）,采取自动的模式，插入Upan盘后自动识别进行挂载
*   **示例**: `/media/linux` 表示挂载的 Linux 文件系统。
*   **备注**: 挂载点由系统动态生成。

#### 4.3 `/mnt` - 临时挂载点

*   **用途**: 提供临时挂载其他文件系统的位置。
*   **示例**: `/mnt/myshare` 用于挂载网络共享或外部设备。
*   **备注**: 常用于手动挂载临时文件系统。

#### 4.4 `/opt` - 可选应用软件包

*   **用途**: 存放通过非包管理器安装的第三方软件。
*   **示例**: `/opt/ORACLE` 表示 Oracle 数据库安装目录。
*   **备注**: 适用于独立安装的商业软件，自己进行手动第三方安装的时候大多选择此文件

#### 4.5 `/usr/local` - 本地用户程序

*   **用途**: 存放本地安装的程序和库文件。
*   **示例**: `/usr/local/bin` 包含本地编译的可执行文件。
*   **备注**: 用于管理员手动安装的软件。

#### 4.6 `/var` - 变量数据

*   **用途**: 存储运行时生成的数据，如日志和缓存。
*   **示例**: `/var/log` 存放系统日志。
*   **备注**: 数据可能随时间增长。

#### 4.7 `/home` - 用户主目录

*   **用途**: 存放用户个人文件和配置。
*   **示例**: `/home/user` 表示特定用户的家目录。
*   **备注**: 每个用户拥有独立的主目录。

#### 4.8 `/root` - 根用户主目录

*   **用途**: 根用户（超级用户）的个人主目录。
*   **示例**: `/root` 包含管理员的配置文件。
*   **备注**: 独立于普通用户主目录。

#### 4.9 `/boot` - 启动文件

*   **用途**: 存储引导加载程序和内核文件。
*   **示例**: 包含 `vmlinuz` 和 `initrd` 文件。
*   **备注**: 用于系统启动过程。

#### 4.10 `/proc` - 进程信息

*   **用途**: 虚拟文件系统，反映系统和进程状态。
*   **示例**: `/proc/cpuinfo` 显示 CPU 信息。
*   **备注**: 不占用磁盘空间，动态生成。

#### 4.11 `/srv` - 服务数据

*   **用途**: 存储服务相关的数据文件。
*   **示例**: `/srv/www` 用于 Web 服务器数据。
*   **备注**: 常用于特定服务的存储。

#### 4.12 `/sys` - 系统信息

*   **用途**: 提供系统硬件和内核信息的虚拟文件系统。
*   **示例**: `/sys/class` 包含设备类信息。
*   **备注**: 基于 Linux 2.6+ 内核。

#### 4.13 `/tmp` - 临时文件

*   **用途**: 存储临时文件，系统重启时清空。
*   **示例**: 用户临时数据存储。
*   **备注**: 安全性较低，建议谨慎使用。

#### 4.14 `/selinux` - SELinux 安全增强

*   **用途**: 存放 SELinux（安全增强型 Linux）相关配置。
*   **示例**: `/selinux` 包含安全策略文件。
*   **备注**: 仅在启用 SELinux 时使用。

#### 4.14 `/etc` - 软件配置文件
- **用途**: 存放全局配置文件

### 5.日志分析

#### 5.1 日志查询相关命令
- **`sudo journalctl -u <service_name>`** : 查询指定服务的日志
    - **`sudo journalctl -u NetworkManager`** ： 查询 **NetworkManager** 相关日志
    - **`sudo journalctl -u polkit`**: 查询polkit相关日志

- **`sudo journalctl -u <service_name> -n <number>`** : 可以限制查收的条数，用于接收最新的日志
    - **`sudo journalctl -u NetworkManager -n 50`**: 查找networkmanager的最近50条最新日志
- **`sudo journalctl -f`** ：实时接收最新的日志来监控

- **`sudo journalctl -b`** : 检测日志在系统启动以后的日志，同时可以针对特定服务
    - **`sudo journalctl -u NetworkManager -b`**: 查询networkmanager自启动以来的日志

- **`sudo journalctl --since "<time>" --until "<time>"`**: 查询特定时间段的日志
    - **`sudo journalctl -u NetworkManager --since "2025-06-15 20:55:00" --until "2025-06-15 21:08:00"`**: 


#### 5.2 日志导出以及摘选
- **`sudo journalctl -u NetworkManager > networkmanager.log`**
    - \> 输出重定向操作符，将前面命令的输出写入文件，而不是显示在终端上
    - networkmanager.log 输出的文件的名称
    - 这里的意思是将NetworkManager的日志输出到名称为networkmanager.log文件夹的里面方便分析

- **`sudo journalctl -u NetworkManager | grep -i error`**: 
    - 摘选出 NetworkManager 的日志中关于error的日志

- **`sudo journalctl -u polkit | grep -i proc`**:
    - 摘选出 polkit 中的proc的日志
    
    

#### 5.3 日志分析技巧

>**6月 15 20:56:25 wenjun-virtual NetworkManager[3434]: <u><warn></u>  [1749992185.8335] <u>error</u> <u>requesting</u> auth for org.freedesktop.NetworkManager.network-control: Authorization check failed: <u>Failed to open file “/proc/3450/status”:</u> No such file or directory,**
-   - Polkit 授权错误（/proc/<pid>/status）是关键问题，导致 nmcli 命令失效。在这里可能是因为NetworkManager的缓存没有清理，也有可能是因为控制NetworkManager的权限不足，3450代表进程端口ID，这个进程受到了限制
    - requesting auth for org.freedesktop.NetworkManager.network-control:  
        - 这里作为整体的错误描述，请求执行network-control 操作的权限失败 
    - Authorization check failed 表示授权失败，没有权限
    - No such file or directory 找不到3450这个进程，文件或目录不存在，可能是遗留缓存的问题也有可能是其他问题

>**6月 15 20:56:36 wenjun-virtual NetworkManager[3434]: <u><info></u>  [1749992196.6243] audit: <u>op="device-managed"</u> <u>arg="managed:1"</u> pid=3463 uid=0 result="success"**
-   - <info> 标签（如 new Ethernet device）确认正常行为，如 ens33 被识别
    - op="device-managed" 这里代表执行操作，为设备托管
    - arg="managed:1" 这里代表托管行为
        - result="success" 则代表托管成功
    -  pid=3463 uid=0 分别代表进程的ID和用户的ID，可以用来分辨不同的进程
    -  [1749992196.6243] 代表精确的时间戳，毫秒级

### 7.电源管理

#### 7.1.systemctl管理电源

##### 7.1.1.相关命令

- **`sudo systemctl poweroff`** : 关机，并且切断电源

- **`sudo systemctl reboot`** : 重启 

- **`sudo systemctl suspend`** : 挂起
    - 将系统状态保存到内存中，并进入低功耗模式。
    - CPU 和其他硬件组件进入低功耗状态，但内存仍然供电。
    - 恢复速度快，但依赖电源（如果断电，数据会丢失）。

- **`sudo systemctl hibernate`** : 休眠
    - 将系统状态保存到硬盘上的交换分区（Swap），然后完全切断电源。
    - 即使断电，也可以恢复之前的系统状态。
    - 恢复速度较慢，因为需要从硬盘加载数据。

- **`sudo systemctl hybrid-sleepe`** : 混合休眠
    - 结合挂起和休眠的功能。
    - 系统状态既保存到内存（快速恢复），也保存到硬盘（防止断电）。
    - 如果不断电，可以从内存快速恢复；如果断电，则从硬盘恢复。

- **`sudo systemctl suspend-then-hibernate`** : 锁定休眠
    - 先挂起一段时间，如果系统在指定时间内没有被唤醒，则自动进入休眠状态。
    - 较为节能

##### 7.1.2.状态查询和支持

- **`systemctl status`**: 查询当前的systemctl模块，看一下是否正常运行

- <u>**`dmesg | grep -i "suspend\|hibernate"`**</u> 用来查看是否有挂起和休眠的日志以此来判断硬件是否支持挂起和休眠

- **`cat /sys/power/state`**: 查看power里面的state文件，这里面通常包含支持的电源状态，比如：
    - **`freeze`**:  冻结模式
        - 进入冻结模式（Freeze Mode），也称为“浅睡眠”。
        - 系统会暂停所有任务，但不会进入深度挂起或休眠。
        - 内存仍然供电，设备保持运行状态，但功耗较低。
        - 唤醒速度快，但是功耗节能有限
    - **`standby`**:  待机模式
        - 相比于冻结模式，关闭更多硬件，功耗降低更多 
    - **`mem`**:  挂起到内存
        - 挂起到内存（Suspend to RAM）。
        - 将系统状态保存到内存中，并关闭大部分硬件组件。
        - 内存仍然供电，因此可以快速恢复。
        - 唤醒速度非常快。
        - 功耗显著降低，但仍依赖电源（如果断电，数据会丢失）。
    - **`disk`**:  休眠
        - 休眠到硬盘（Hibernate to Disk）。
        - 将系统状态保存到硬盘（通常是交换分区），然后完全切断电源。
        - 即使断电，也可以恢复之前的系统状态。
        - 唤醒速度较慢（需要从硬盘加载数据）。功耗最低。


#### 7.2.传统电源管理

- **`sudo halt`**:    停止操作系统的核心运行，但不一定会切断电源。

- **`sudo reboot`**:  重新启动系统。

- **`sudo poweroff`**: 直接切断电源，完全关闭系统。

- **`sudo shutdown now`**:  立即关机。

- **`shutdown +5 "System will shutdown in 5 minutes"`**: 计划 5 分钟后关机，并显示消息。

- **`sync`**: 同步内存内容到磁盘里面
    - 大多电源相关命令会自动提交该指令，可以手动调用，保险起见

### 9.用户管理

#### 9.1.用户登录和注销

##### 9.1.1.Root权限登录

- - 由于ubuntu系统的原因，在进行root相关的操作时，全部需要 **`sudo`** 的前缀来提高权限

>  一个系统里面会有多个用户，不同的用户对应的环境变量不一样即用户的主目录不一样，此时你配置的一些主目录里面的配置文件在切换到其他用户时就不会生效，例如 **`vimrc`** 配置文件
>>    普通用户的 Vim 配置文件：/home/user1/.vimrc
>>    root 用户的 Vim 配置文件：/root/.vimrc
>>>   你也可以创建符号链接，将root用户的vimrc文件指向普通用户的vimrc文件

- **`sudo su - root`**: 完全转换到root用户，此时加载了root 用户的完整环境变量
    - <u>主目录从 /home/username 变为 /root。</u>
    - 某些软件可能会加载 /root/.config 或 /root/.bashrc 等配置文件，而不是普通用户的配置文件。

- **`sudo su - root`**: 临时提供root权限，仅切换用户身份，保留当前用户的环境变量和工作目录。可用权限较低


> 对于需要全局配置的文件可以选择将文件配置到etc里面，etc是作为全局的配置文件夹，但是需要root权限操作。系统目录的其他文件也是全局的

- 对于root后的账户，它的终端所在的工作目录会变更
    - **`/home/username`**: 为普通用户的默认工作目录
    - **`/root`**: 为root用户的默认工作目录
        - **`cd ~`**: 切换到默认工作目录
        - **`cd x `**: 切换到x文件夹作为工作目录

        - 区别于windows的注销，直接到用户输入pin的登录界面
        -  无法强制断开其他用户
    -  还可以用于从root账号回到用户账号，即返回上一级环境。

#### 9.4.删除用户账号

##### 9.4.1.确认和终止用户进程

- **`who`**: 此命令用查看当前有哪些用户登录到系统，并监控活动会话。

- **`ps aux | grep pts/ | grep username`**；确认用户是否有活动会话或进程，如有重要进程，则需要进行处理，如果没有直接终止所有进程再进行安全删除
    - 通过上面这个命令得到PID，然后通过 **`kill PID`** 来直接杀死进程也可以终止进程
    - **`ps aux`**: 这个命令会列出系统中所有的进程
        - **`ps`** 是 "process status" 的缩写，用于显示当前系统中运行的进程。
        - aux 是一组选项：
            - a: 显示所有用户的进程（包括其他用户的）。
            - u: 以用户可读的格式输出（包含用户、CPU、内存等信息）。
            - x: 显示没有控制终端的进程。
    - **`|`** :管道符号，用于将前一个命令的输出作为后一个命令的输入。
    - **`grep pts/`** :  这一步过滤出与终端会话相关的进程。
        - **`grep`** 是 "global regular expression print" 的缩写，用于搜索文本中的匹配行。
        - **`pts/`** 是伪终端（pseudo-terminal）的标志，表示与终端相关的会话。
    - **`grep username`** :再次使用 grep，筛选出属于指定用户（username）的进程。
    
- **`pkill -KILL -u username`**: 用于一次性终止指定用户的所有进程，适合批量操作。
    - **`pkill`** 是一个更高级的工具，可以根据进程名、用户等条件直接终止进程。
    - **`-KILL:`** 强制终止信号，等同于 kill -9。

#### 9.5.用户数据查询

        - **`drwxr-xr-x`**：这是文件或目录的权限和类型信息，分为以下几个部：
            - **`d`** : 表示这是一个目录 （directory）。如果是普通文件，则这里是 -；如果是符号链接，则是 l。

            - **`r-x`** : 这是其他用户的权限（即既不是文件所有者，也不属于文件所属组的用户）：

#### 9.7.用户小组

##### 9.7.1.组间联系

用户的组分为 主组（Primary Group）和附加组（Secondary Group）
- 主组
    - 每个用户都有一个默认的主组，用户创建文件或目录时默认使用的组。

    - 唯一性 : 每个用户只能有一个主组。
    - 默认归属 : 用户创建的文件或目录会自动继承主组的权限。
    - 存储位置 : 主组信息存储在 /etc/passwd 文件中，与用户名直接关联。
    - <u>主组使用</u>
        - 主组通常用于管理用户创建的文件或目录的默认权限，在团队协作中，可以将所有用户的主组设置为同一个组（如 developers），以便他们共享文件。

- 附加组
    - 附加组是用户除主组之外所属的其他组，用于扩展用户的权限范围。
    
    - 多对一 : 一个用户可以属于多个附加组。
    - 权限扩展 : 用户可以通过附加组获得额外的权限（如访问特定设备、运行特定服务等）。
    - 存储位置 : 附加组信息存储在 /etc/group 文件中。
    - <u>附加组使用</u>
        - 附加组不会影响文件的默认组，但会影响用户的权限范围。可以分配不同用户不同权限

##### 9.7.2.组的添加和修改

- 修改主组
    - **`sudo usermod -g 主组名称 username`**:	更改用户的主组（GID），不会影响附加组。 


- 修改附加组
    - **`sudo usermod -aG 附加组名称 username`**:	用于添加用户到附加组，同时保留原有的附加组以及主组 
    - **`sudo usermod -G 附加组名称 username`**：添加用户到附加组，并且解除原有的所有附加组的资格，但是不影响主组

- 新建组
    - **`sudo groupadd groupname`**: 新建一个组，groupname为新建组的名字
    
    - **`grep groupname /etc/group`**: 检查组的配置文件，验证组是否建立成功
    - 这个组是可以作为主组也可以作为附加组，取决于你如何添加，比如 **`-g`** 就是添加到主组，那么此时新建的组作为主组，而 **`-aG`** 则相反，作为附加组

### 11.权限管理


#### 11.1.权限表示方法

##### 11.1.1.权限符号含义

- 读权限（**`r`**） ：
    - 对文件：允许查看文件内容。
    - 对目录：允许列出目录中的文件和子目录。
- 写权限（**`w`**） ：
    - 对文件：允许修改文件内容或删除文件。
    - 对目录：允许在目录中创建、删除或重命名文件。
- 执行权限（**`x`**） ：
    - 对文件：允许将文件作为程序运行。
    - 对目录：允许进入目录并访问其内容。
- **`-`**:代表无权限

##### 11.1.2.文件权限解析

- 这里对于一个新创建的文件的组默认是属于用户的主组，那么对于这个主组的所有用户都拥有读和执行的权限，如果这个文件被移动到其他组，那么此时主组的用户也是属于其他用户，而被移动到的那个组的用户拥有组的权限

###### 符号表示法：

- **`-rwxr-xr--`**: 
    - >第一位 - 表示这是一个普通文件（如果是目录则是 d）。
        >**接下来的每三位分别表示所有者、组和其他用户的权限** 
    - 这一行权限代表的是这个文件的所有者支持<u> 读 、 写 、 执行</u>权限，同时这个文件归属的组则支持<u> 读 、 执行</u>权限，但不允许写，其他用户则只支持<u> 读</u>权限



###### 数字表示法

- 每种权限可以用数字表示：r = 4，w = 2，x = 1
    - **`rwx`** = 4 + 2 + 1 = 7
    - **`r-x`** = 4 + 0 + 1 = 5
    - **`r--`** = 4 + 0 + 0 = 4
    - 示例：754 表示所有者有 rwx（7），组有 r-x（5），其他用户有 r--（4）。


#### 11.1.查看和修改权限


- **`ls -l`**: 查看该文件的属性，包含权限，文件所有者，文件所属的组

##### 11.1.1.修改操作

- **`chmod [who][operator][permissions] file`**: 更改文件或目录的访问权限。
    - who：指定权限的作用对象：
        - u：所有者（User）
        - g：所属组（Group）
        - o：其他用户（Others）
        - a：所有人（All，等价于 ugo）
    - operator：操作符：
        - +：添加权限
        - -：移除权限
        - =：设置权限
        使用实例：**`chmod u+x Hello.c`** 为Hello.c文件的所有者添加可执行的权限
        或者： **`chmod 754 Hello.c`** 即权限设置为 **`rwxr-xr--`**
    - 这里仅仅是文件所有者才能修改，即使是采用sudo提高权限也可能失败

- **`chown [options] [owner][:group] file...`**:用于更改文件或目录的所有者和所属组(需要 sudo 权限)
    - owner：新的所有者用户名或 UID。
    - group：新的所属组名或 GID（可选）。
    - file...：目标文件或目录。
    - **` : `**: 如果仅仅更改组也要加上
    - 常用选项
        - **`-R`**: 递归更改子文件和子目录 
    - 使用示例：**`sudo chown new_wenjun:new_wenjun Hello.c`**
        - 将该文件的所有着更改为用户new_wenjun，所属组也更改为用户new_wenjun的主组
        


#### 11.1.2.特殊权限


###### SGID（Set Group ID）

- SGID 是一种特殊的权限位，用于控制文件或目录的行为。
    - 对目录 ：在该目录中创建的新文件会自动继承目录的所属组，而不是创建者的主组。
    - 对文件 ：运行该文件的用户会临时获得文件所属组的权限。从而可以访问该组的资源。
- **`sudo chmod g+s /shared`**: g+s 表示为文件或目录设置 SGID 位。
    - 设置完后，可以通过 ls -l来查看文件或者目录的权限，返回的权限 **`-rwxr-sr-x`** 第六位出现 <u>**`s`**</u> 代表设置SGID成功，且有执行权限，如果大写的S，则代表没有执行权限，但是仍然设置成功


###### SUID（Set User ID）

- 当设置了 SUID 位时，运行该文件的用户会临时以文件所有者的身份运行，而不是以自己的身份运行。
- 运行该文件时，进程会临时以文件所有者的身份运行

- **`sudo chmod u+s /shared`**: u+s 表示为文件或目录设置 SUID 位。
    - 设置完后，可以通过 ls -l来查看文件或者目录的权限，返回的权限 **`-rwxr-sr-x`** 第四位出现 <u>**`s`**</u> 代表设置 SUID 成功，且有执行权限，如果大写的S，则代表没有执行权限，但是仍然设置成功

### 13.进程以及线程

#### 13.1.相关概念

- 进程是程序的一次运行实例，包含独立的地址空间（代码段、数据段、堆栈）,而在一次进程里面可以创建多个线程，是进程内部的执行单元，共享进程的地址空间和资源（如内存、文件描述符），但有独立的栈和寄存器。

##### 13.1.1.进程

- 进程:

  - 每个进程都有唯一的**`PID`**(process id )，同时在该进程下面可以生成子进程，而PPID为你父进程的PID，下面的结构图可以展示
    - **`systemd (PID=1)`**
       └── **`bash (PID=1234, PPID=1)`**
           └── **`vim (PID=5678, PPID=1234)`** 
  - **`pstree`**: 显示当前的进程的结构树形式

  - 进程的状态分为 : **就绪**、**运行**、**阻塞**、**僵尸**、**孤儿**
  - **`ps aux`**: 这个命令会列出系统中所有的进程
    - **`ps`** 是 "process status" 的缩写，用于显示当前系统中运行的进程。
    - **`aux`** 是一组选项：
      - a: 显示所有用户的进程（包括其他用户的）。
      - u: 以用户可读的格式输出（包含用户、CPU、内存等信息）。
      - x: 显示没有控制终端的进程。

##### 13.1.2.线程

- 线程:
  - **`ps -eLf`**: 输出系统中所有进程及其线程（LWP）的信息
    - **`-e`**: 显示所有进程
      **`-L`**: 显示线程（Light Weight Process / LWP）
      **`-f`**: 完整格式（显示 UID, PID, PPID, LWP 等详细信息）
  - **`ps -T -p <pid>`**: 输出某个PID进程的所有线程
    - **`-T`**: 显示指定进程的所有线程（Threads）
    - **`-p <pid>`**: 指定PID进程
  - **`LWP`**: 每个线程之间都有的唯一线程ID
    - 当一个进程中只有一个线程时，LWP = = PID
  - 相比于进程，它的创建只需分配栈和线程控制块（TCB），开销较小，速度更快。同时线程之间切换速度也更加快

##### 13.1.3.调度机制概况

- 基于**CFS**(完全公平调度器)调度机制的优先级影响
  - nice 值，用户层面可以修改，间接影响进程优先级(用户一般无法直接修改程序优先级)
  - nice值转化为优先级的公式（近似）: **`优先级 (PR) = 100 + nice 值`**
- 基于实时调度机制
  - 需要打实时补丁，优化内核任务调度
  - root 权限直接设置优先级(0-99)

> **针对嵌入式开发一般是创建进程，即创建一个进程来统一管理多个线程，每个线程对应你设置的任务，然后囊括多个线程，内核调度的最小单位也就是线程**



#### 13.2.调度机制详解

##### 13.2.1.实时调度策略详解和分析

- 实时系统的优先级(0-99)高于GFS(完全公平调度)调度策略的线程优先级(nice值间接控制)

- **SCHED_FIFO**：高优先级任务持续运行，直到阻塞、结束或被更高优先级任务抢占，无时间片限制。
  - 先进先出的实时调度策略；
  - **同优先级的进程按 FIFO 顺序执行；**   同时可以结合信号量来保证两个任务都执行完毕后执行下一个优先级的任务，或者其他的任务，但是这需要信号量来控制任务的阻塞以及让出
    - 线程任务中间可以通过信号量随时阻塞等待另一个任务执行
  - 高优先级进程会抢占低优先级进程；
  - 不会自动释放 CPU，除非主动让出，一般由信号量来控制，类似于不开启时间片轮转的RTOS任务调度。
- **SCHED_RR**：同优先级任务轮流运行，每个任务有时间片（默认约 100ms），适合多个任务共享 CPU。
  - 时间片轮转的实时调度策略；
  - 同优先级的进程轮流执行，时间片耗尽后让出 CPU；当然任务时间小于时间片可以提前执行完去切换其他任务
  - 高优先级进程仍可抢占低优先级进程；
  - 类似 RTOS 中的任务调度。

- **SCHED_DEADLINE** : 基于截止时间（Deadline）的调度策略；为任务分配运行时间（Runtime）、截止时间（Deadline）和周期（Period）；
  - 基于截止时间（Deadline）的调度策略(不看优先级)；每次调度时，选择绝对截止时间最早的任务运行。
  - 必须满足 Σ (runtime / period) < 1（单核），确保所有任务可调度。
  - 为任务分配运行时间（Runtime）、截止时间（Deadline）和周期（Period）；
  - 适合硬实时任务（如音视频同步），或者固定周期开启某个任务；
  - **不能与其他实时调度策略共存** ，必须绑定在一个隔离的CPU上面，在这颗CPU上面只能是这种策略的任务,不与其他调度策略线程混合运行；<u>所以他和其他实时调度的线程必须通过进程间通信，无法进行线程通信</u>
- 如果需要硬实时，需要打实时补丁，让系统处于RTOS的状态
- **混用思考：** **SCHED_FIFO**和**SCHED_RR**可以共存，可以通过信号量来决定不同线程的任务执行顺序(如果两个同等优先级的线程，任务调度策略都为**SCHED_FIFO**，那么执行顺序就按照FIFO顺序执行，这种情况和**SCHED_RR**的处理差不多，只是多了个时间片，可以防止意外情况)，同等优先级下我任务作用不大，但是如果是不同优先级，比如FIFO通过信号量让出了CPU，再另一个优先级上面，两个线程任务通过SCHED_RR调度可以实现多任务进程轮流执行,但是一般不会这么操作，全部改用**SCHED_FIFO**更加好管理

##### 13.2.2.调度策略的使用思考

- 针对硬实时任务，比如固定10ms的PID控制，那么这个线程的调度策略就设置为 **SCHED_DEADLINE** ，这里可以设置强周期运行，类似于定时器中断，然后针对都是10ms周期的任务，剩下的可以使用互斥锁或者其他信号量来控制周期任务的先后顺序，或者说另外一个线程就设置为
- 如果你主动让出CPU的使用权，那么需要配合信号量唤醒否则下一次就无法执行，比如一个高优先级的任务执行完后可以主动让出CPU，然后其他低优先级任务处理完后可以再次唤醒这个任务来抢占调度。
  - Linux开发板一般不设置定时器
  - 通常对于调度策略为 **SCHED_DEADLINE** 的线程，为了保证效率可以在任务提前完成的时候主动让出cpu。
- 对于<u>**多核**</u>而言，你需要将每一个线程绑定到一个核心减少任务调度的开销，增加实时性，但是这里只针对  **SCHED_FIFO**  和 **SCHED_RR** 这两个调度策略，<u>**SCHED_DEADLINE** 是针对多核全局调度</u>

##### 13.2.3.任务调度的走势

- STAT：进程状态（Process Status）

| 状态 | 说明                                                         |
| ---- | ------------------------------------------------------------ |
| R    | 运行状态。严格来说，应该是“可运行状态”，即表示进程在运行队列中，处于正在执行或即将运行状态，只有在该状态的进程才可能在 CPU 上运行，而同一时刻可能有多个进程处于可运行状态。 |
| S    | 可中断的睡眠状态。处于这个状态的进程因为等待某种事件的发生而被挂起，比如进程在等待信号。 |
| D    | 不可中断的睡眠状态。通常是在等待输入或输出（I/O）完成，处于这种状态的进程不能响应异步信号。 |
| T    | 停止状态。通常是被shell的工作信号控制，或因为它被追踪，进程正处于调试器的控制之下。 |
| Z    | 退出状态。进程成为僵尸进程。                                 |
| X    | 退出状态。进程即将被回收。                                   |
| s    | 进程是会话其首进程。                                         |
| l    | 进程是多线程的。                                             |
| +    | 进程属于前台进程组。                                         |
| <    | 高优先级任务。                                               |

- 调用进程的本质是调用进程里面的线程，所有进程通常是作为一个统一的容器，所以你所设置的任务调度策略可以是具体到针对每个线程的
- 对于多核的 **`mpu`** ，你需要确保有一颗核心没有绑定任何用户的指定任务，减少系统运行风险，以四核为例，三颗核心可以绑定多个任务，即用户指定的高级任务，对于强实时性的任务你也可以直接使用 **SCHED_FIFO** 来进行抢占式任务调度，配合信号量完成任务切换，对于非强实时任务你也可以通过普通任务调度或者其他空闲任务调度(**`SCHED_IDLE`** ) 

##### 13.2.4.实时查看任务进程的状态

- 写 **`shell`** 脚本：**`shell`** 脚本是系统根据各个核心的负载情况灵活调度，无需指定核心，可以作为系统任务

  - ```shell

        echo "Time: $(date)"

##### 13.4.1.线程基础使用和创建

- **代码解析：**

  ```c
  #include <pthread.h>   // POSIX 线程库
  #include <stdio.h>     // 标准输入输出（printf）
  #include <stdlib.h>    // 标准库函数（exit, perror）
  #include <unistd.h>    // sleep(), getpid() 
  
  // 子线程的函数
  void* thread_func(void* arg) {
      int id = *(int*)arg;
      // %lu 为 unsigned long 的占位符
      printf("Thread %lu (ID %d) running\n", (unsigned long)pthread_self(), id);
      sleep(1);  // 延时1s usleep(n)是微妙级别精度更高
      return NULL;
  }
  
  int main() {
      pthread_t threads[2];
      int ids[2] = {1, 2};
  
      printf("Main thread TID: %lu, PID: %d\n", (unsigned long)pthread_self(), getpid());
  
      for (int i = 0; i < 2; i++) {
          if (pthread_create(&threads[i], NULL, thread_func, &ids[i]) != 0) {
              perror("pthread_create");
              exit(EXIT_FAILURE);
          }
      }
  
      for (int i = 0; i < 2; i++) {
          pthread_join(threads[i], NULL);
      }
      printf("Main thread exiting\n");
      return 0;
  }
  ```

1. **子线程函数格式：**

- **`void* thread_func(void* arg)`** ：这是一个需要被创建的子线程，一般任务分类就按照这种子线程即可

  - 同时作为子线程，它的格式一定要满足这种格式，可以修改命名，但是整体格式不能变
  - **`void*`** ： 表示一个任何类型的指针，这里定义的子线程要满足返回的是一个指针，可以是任意类型，同时传入的数据也必须是<u>任意数据类型的指针</u>

- **`int id = *(int*)arg;`** ： 由于任务传入的指针为任意类型，实际应用必须转化为具体的类型指针，这里首先将指针转化为 **`int *`** 类型，再通过 ***** 解引用

- **` return NULL;`** ： 任务返回一个空指针，<u>**但其实你可以返回任何你想要返回的处理后的数据**</u>

  - ```c
    void* thread_func(void* arg) {
        int value = *(int*)arg;
        int* result = malloc(sizeof(int));   // 动态分配堆内存，注意最后需要 free(result)释放内存
        *result = value * 2;
        return result;  // 返回一个指针
    }
    ```

  - 以这个代码为例，先用 malloc创建一块堆内存，**`int* result = malloc(sizeof(int));`** 然后把处理后的数据放进去，然后返回这个数据的指针

  - 然后你可以使用一个函数去获取到这个线程的函数返回数据

  - **`int pthread_join(pthread_t thread, void **retval);`** 这是函数原型，**`thread`** 为你的线程ID，通过 **`pthread_create`** 可以获得你的线程ID，**`void **retval`** 则是一个储存的指针类型变量，需要提前创建然后开始接收数据，它会等待线程结束然后返回数据  <a id="线程清理"></a>

    - ```c
      void* retval; // 首先创建一个任意类型的指针变量
      pthread_join(thread, &retval); // 按照线程ID，将线程的返回值提取出来
      ```

2. **线程创建和执行函数格式：**

- ```c
  int pthread_create(pthread_t *thread, 
                     const pthread_attr_t *attr,
                     void *(*start_routine) (void *), 
                     void *arg);
  ```

  - 这是一个子线程的 “启动键”，会给他传入参数，同时引出一个数据返回的接口，并且启动它，子线程开始运行
  - **`*thread`** : 这是一个参数返回的接口，它是用来返回线程ID(TID)的,每个线程都有自己的编号，方便提取线程的返回数据
  - **`*attr`** : 线程属性，对于 Linux 系统里面的实时调度策略，默认为NULL，采用公平调度器
  - **`start_routine`** : 线程入口函数
  - **`*arg`** : 传给线程函数的参数

3. **线程错误处理** :

- **`exit(EXIT_FAILURE)`** ：退出和终止当前的整个进程，并且告诉计算机产生了错误
- **`echo $?`** : 通过这个命令可以查看错误的原因

4. **线程清理**：

- **`pthread_join(thread, &retval)`** ：[详细说明](#线程清理)
  - 等待线程结束
  - 接收返回值
  - 清理线程资源
  - 如果不允许这个函数，可能会出现僵尸程序，内部没有释放，占用程序资源

5. **实时调度策略配置：**

> 线程不设置实时调度，那么就是系统随机调度

- ```c
  #include <pthread.h>
  #include <stdio.h>
  #include <stdlib.h>
  #include <sched.h>
  #include <unistd.h>
  
  void* thread_func(void* arg) {
      int count = 0;
      while (1) {
          printf("实时线程运行中... %d\n", count++);
          usleep(500000);  // 每 0.5 秒执行一次
      }
      return NULL;
  }
  
  int main() {
      pthread_t thread;
      pthread_attr_t attr;
      struct sched_param param;
  
      // 初始化属性
      pthread_attr_init(&attr);
  
      // 设置为 SCHED_FIFO 实时调度策略
      pthread_attr_setschedpolicy(&attr, SCHED_FIFO);
  
      // 设置优先级为 50（1~99）
      param.sched_priority = 50;
      pthread_attr_setschedparam(&attr, &param);
  
      // 显式继承调度策略
      pthread_attr_setinheritsched(&attr, PTHREAD_EXPLICIT_SCHED);
  
      // 创建线程
      if (pthread_create(&thread, &attr, thread_func, NULL) != 0) {
          perror("pthread_create");
          exit(EXIT_FAILURE);
      }
  
      // 销毁属性
      pthread_attr_destroy(&attr);
  
      // 主线程等待
      pthread_join(thread, NULL);
      return 0;
  }
  ```

  - **`pthread_attr_t attr;`** ：是一个用于保存线程属性的结构体；
  - **`pthread_attr_init(&attr);`** ：对于这个线程属性结构体再修改之前必须首先要初始化
  - **` pthread_attr_setschedpolicy(&attr, SCHED_FIFO);`** : 设置调度策略，即线程属性
    - **SCHED_FIFO**   [详细描述](#13.2.1.几种实时调度策略)
    - **SCHED_RR**
    - **SCHED_DEADLINE**
  - **`struct sched_param param;`** 这是一个结构体用来封装调度参数(比如优先级)
    - **`param.sched_priority = 50;`** 
  - **`pthread_attr_setschedparam(&attr, &param);`** 将优先级和线程属性绑定，这一步是必须的，否则优先级不会生效。
  - **`pthread_attr_setinheritsched(&attr, PTHREAD_EXPLICIT_SCHED);`** ：告诉系统不要从父线程继承调度策略，默认情况下，线程会继承父线程的调度策略，修改后必须加上这句
  - **`pthread_create(&thread, &attr, thread_func, NULL);`** : 最后启动线程的时候将属性应用上去 **`&attr`** 
  - **`pthread_attr_destroy(&attr);`** : 使用完属性结构体后销毁掉，释放资源

##### 13.4.2.线程同步机制

###### **互斥锁(防止资源竞争)：**  

- 保证读写公共资源的同步锁，能够防止资源竞争，同时相比**读写锁**，不支持多线程同时读取

```c
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
// 初始互斥锁，并且默认属性，但是容易造成优先级反转
//pthread_mutex_t mutex;
//pthread_mutex_init(&mutex, NULL); // 初始化锁的属性，NULL代表默认属性，可以选择

// 定义一个锁
pthread_mutex_t mutex;

int shared_counter = 0;
void* thread_func(void* arg) {
    int id = *(int*)arg;
    for (int i = 0; i < 5; i++) {
        // 访问共享资源前调用的加锁
        pthread_mutex_lock(&mutex);
        shared_counter++;
        printf("Thread %lu (ID %d) counter: %d\n", (unsigned long)pthread_self(), id, shared_counter);
        pthread_mutex_unlock(&mutex);   // 访问结束后释放锁，如果使用了递归锁，则需要根据锁的次数释放
        // 没有让出cpu，一个线程抢占到锁即使释放也会继续抢占，可以通过信号量或者条件变量来调整执行顺序
        // 两个线程随机抢占锁
//        usleep(100000); // 100ms
    }
    return NULL;
}

int main() {

    // 动态初始化互斥锁，支持优先级继承，不会造成优先级反转，比如低优先级持有锁造成高优先级任务阻塞
    pthread_mutexattr_t mutex_attr;  // 首先创建一个锁的属性结构体实例
    pthread_mutexattr_init(&mutex_attr);  // 初始化锁的属性结构体实例
    pthread_mutexattr_setprotocol(&mutex_attr, PTHREAD_PRIO_INHERIT); // 设置锁的属性为优先级继承
    pthread_mutex_init(&mutex, &mutex_attr);  // 将锁的属性赋予给创建的互斥锁

    pthread_t threads[2];
    int ids[2] = {1, 2};

    printf("Main thread TID: %lu, PID: %d\n", (unsigned long)pthread_self(), getpid());

    for (int i = 0; i < 2; i++) // 连续开启两个独立线程任务，传入不同的id，并发的访问资源
    {
        // 默认的线程属性，遵循公平原则，线程之间并发运行，可能会出现轮流执行的情况，但是实际上是谁抢到锁就会开始执行
        if (pthread_create(&threads[i], NULL, thread_func, &ids[i]) != 0)
        {
            perror("pthread_create");
            exit(EXIT_FAILURE);
        }
    }

    for (int i = 0; i < 2; i++) {
        pthread_join(threads[i], NULL);  // 清理线程资源，NULL代表无需接收返回值
    }
    printf("Main thread exiting, final counter: %d\n", shared_counter);

    pthread_mutex_destroy(&mutex);          // 销毁锁
    pthread_mutexattr_destroy(&mutex_attr);  // 销毁锁的属性结构体实例

    return 0;


}
```

- **`pthread_mutex_t mutex;`** 创建一个锁的实例

- ```c
  // 动态初始化互斥锁，支持优先级继承，不会造成优先级反转，比如低优先级持有锁造成高优先级任务阻塞
  pthread_mutexattr_t mutex_attr;  // 首先创建一个锁的属性结构体实例
  pthread_mutexattr_init(&mutex_attr);  // 初始化锁的属性结构体实例
  pthread_mutexattr_setprotocol(&mutex_attr, PTHREAD_PRIO_INHERIT); // 设置锁的属性为优先级继承
  pthread_mutex_init(&mutex, &mutex_attr);  // 将锁的属性赋予给创建的互斥锁
  ```

- **`pthread_mutex_lock(&mutex);`** 在访问竞争资源的适合提前调用

- **`pthread_mutex_unlock(&mutex);`** 访问完竞争资源后需要释放锁

  - 如果使用**递归锁**，必须将嵌套的多层锁必须逐一解开，一次只能解一把，<u>**普通锁不能一直加锁，会造成死锁**</u>

  - ```c
    // 设置为递归锁
    pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE);
    ```

  - 配合继承优先级的属性，一起设置到锁的属性结构体里面

- ```c
  pthread_mutex_destroy(&mutex);          // 销毁锁
  pthread_mutexattr_destroy(&mutex_attr);  // 销毁锁的属性结构体实例
  ```

- 最后将锁销毁，同时将锁的属性结构体也销毁

- <u>**特殊使用：**</u>

  - 非阻塞启动读取锁：**`pthread_mutex_trylock`** 
  - 没有抢到锁自动返回 **`EBUSY`** ，成功则返回 1 **<u>允许线程执行替代逻辑</u>** 



###### **读写锁(防止资源竞争)：**

- 读写锁也是一种多线程同步机制，它允许多个线程同时读取共享资源 ，但写入时必须独占资源 。<u>不支持递归读写锁,不建议优先级继承</u>读写锁的读和写为同一把锁，一般情况下都适用，能提高并发读取的效率，但是写锁应用场景在操作不频繁的情况下，比如写频率在50ms以上
- **且一般情况下不允许读写锁和互斥锁混用，否则会造成锁竞争**

```c
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

//pthread_mutex_t rwlock = PTHREAD_MUTEX_INITIALIZER;
// 获取读写锁，并且以默认属性直接初始化
//pthread_rwlock_t rwlock = PTHREAD_RWLOCK_INITIALIZER;

// 创建读写锁
pthread_rwlock_t rwlock;

int shared_data = 0;
void* reader(void* arg) {
    int id = *(int*)arg;
    // 调用读写锁在读取或者写入公共资源的时候，和互斥锁的区别就是读写锁支持更多并发访问
    pthread_rwlock_rdlock(&rwlock);
    printf("Reader %d: shared_data = %d\n", id, shared_data);
    sleep(1); // 模拟读取时间
    // 释放读写锁
    pthread_rwlock_unlock(&rwlock);
    return NULL;
}

void* writer(void* arg)
{
    // 写线程必须等待所有读线程释放锁后才能进入 ,即这个写线程也需要等待读线程的锁释放掉
    pthread_rwlock_wrlock(&rwlock);
    shared_data++;
    printf("Writer: shared_data = %d\n", shared_data);
    sleep(1); // 模拟写入时间
    pthread_rwlock_unlock(&rwlock);
    return NULL;
}

int main()
{
    // 初始化读写锁
    pthread_rwlock_init(&rwlock, NULL);

    pthread_t readers[3], writer_thread;
    int ids[3] = {1, 2, 3};

    // 启动第一个读线程
    pthread_create(&readers[0], NULL, reader, &ids[0]);
    usleep(500000);  // 延迟 0.5 秒

    // 再启动写线程
    pthread_create(&writer_thread, NULL, writer, NULL);
    usleep(500000);  // 延迟 0.5 秒

    // 再启动其他两个读线程
    pthread_create(&readers[1], NULL, reader, &ids[1]);
    pthread_create(&readers[2], NULL, reader, &ids[2]);

    for (int i = 0; i < 3; i++) {
        pthread_join(readers[i], NULL);
    }
    pthread_join(writer_thread, NULL);

    pthread_rwlock_destroy(&rwlock);
    return 0;
}
```

- 写操作需要等待读操作释放锁，同时一旦写操作抢到读写锁，读操作就必须等待读写锁的释放，但是读操作拿到锁，那么读操作可以并发进行，上述的读线程是加入了 **`sleep(1); `** ,且没有释放锁的前提下，两个读操作都通过读取锁并发进行了读取，效率高
- 非阻塞启动读取锁：**`pthread_rwlock_tryrdlock`**  
  - 没有抢到锁自动返回 **`EBUSY`** ，成功则返回 1 **<u>允许线程执行替代逻辑</u>** 
- 传统启动读取锁：  **`pthread_rwlock_rdlock`** 

###### 信号量(任务执行顺序)：POSIX信号量(线程)

- **`sem_t`**（POSIX 信号量）一般用于<u>线程间同步</u>
- 信号量分为二值信号和计数信号量，如果设定这个信号量最多允许一个线程访问，那么它就是二值信号量，如果可以有多个，则是计数信号量

```c
#include <pthread.h>
#include <semaphore.h>
#include <stdio.h>
#include <unistd.h>

// 创建信号量实例
sem_t semA, semB;

void* threadA(void* arg) {
    while (1)
    {
        // 信号量减1，如果semA本身就是0，那么就会阻塞
        sem_wait(&semA);
        printf("Thread A running\n");
        // 给semB发送信号，semB 信号量加1
        sem_post(&semB);
    }
    return NULL;
}

void* threadB(void* arg) {
    while (1)
    {
        // semB信号量减1，确保信号量为1及以上，初始的时候 semB为0，必须等threadA任务发送信号量，此时semB=1
        sem_wait(&semB);
        printf("Thread B running\n");
        sem_post(&semA);
    }
    return NULL;
}

int main() {
    pthread_t t1, t2;

    // 初始化信号量 
    sem_init(&semA, 0, 1);  // 初始值为 1
    sem_init(&semB, 0, 0);  // 初始值为 0

    // 创建两个线程
    pthread_create(&t1, NULL, threadA, NULL);
    pthread_create(&t2, NULL, threadB, NULL);

    // 等待线程结束（示例中是无限循环，所以不会结束）
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);

    // 销毁信号量
    sem_destroy(&semA);
    sem_destroy(&semB);

    return 0;
}
```

- **`int sem_init(sem_t *sem, int pshared, unsigned int value);`**: 信号量初始化
  - **`pshared`** : 0代表线程间通信，1代表进程间通信(不是所有平台支持)，跨进程一般使用 System V信号量
  - **`value`** : 初始的信号量值，初始的时候就决定这个信号量可以容纳多少个线程通信，0或者1直接当作二值信号量，只能容纳一个线程通信，
- **`int sem_wait(sem_t *sem);`** 等待信号量
  - 信号量减 1 ，如果无法减1，则会阻塞线程

- **`int sem_post(sem_t *sem);`**  发送信号量
  - 信号量加1，唤醒一个等待的线程
- 信号量可以叠加使用，完成多个任务的执行顺序操控
- **`int sem_destroy(sem_t *sem);`** 程序执行完毕摧毁信号量

### 16.通过APT下载文件目录获取以及图标启动修改

#### 16.1.安装可调节文字大小软件

```bash
# 举例，如果通过apt下载的软件如何获取它的文件目录，以及设置图标启动
sudo apt install gnome-tweaks
```

```
# ✅ 查找某个文件属于哪个已安装包
dpkg -S /usr/bin/gnome-tweaks

dpkg -L gnome-tweaks | grep bin

# ✅ 查找刚安装的软件的可执行文件位置
which gnome-tweaks

whereis gnome-tweaks
```

**获取的目录类型说明：**

- `/usr/bin/gnome-tweaks`：可执行程序（启动用）

- `/usr/share/applications/gnome-tweaks.desktop`：默认的桌面启动项配置文件

- `/usr/share/icons/`：软件图标文件（通常在这个目录下的 hicolor 子目录）

- `/usr/share/gnome-tweaks/`：软件的资源 / 配置文件目录

#### 16.2.针对用户来编写启动项(非root)

```bash
# 创建启动项文件
vim ~/.local/share/applications/org.gnome.tweaks-custom.desktop
```

```bash
# 启动项文件编写
[Desktop Entry]                       # 固定开头，标识这是一个桌面启动项文件（所有 .desktop 文件都必须写）
Name=GNOME 调整工具                    # 定义的应用名称
Comment=Customize GNOME settings     # 应用描述，鼠标悬停时显示，可自定义；
Exec=/usr/bin/gnome-tweaks           # 可执行文件路径，必须严格正确，可以通过dpkg -L gnome-tweaks 查找相关目录
Icon=/home/wenjun/APP_logo/wenzi_change_logo.png    # 应用图标logo路径
Terminal=false      # 表示启动程序时不弹出终端窗口（图形界面程序都写 false，命令行程序写 true）；
Type=Application    # 固定值，标识这是一个应用程序启动项（还有 Link/Directory 等类型，极少用）；
Categories=Settings;GTK;GNOME;Utility;      # 用于应用搜索和菜单分类，可以参考程序自己的定义，也可自定义
Keywords=GNOME;tweaks;设置;调整;              # 搜索关键词，方便在应用列表里搜到程序，可自定义；
```

- 本质上是创建一个启动项，并且内部直接指向程序的可执行文件
