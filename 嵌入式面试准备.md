**协议规定：**

- **只有 SCL 低电平时，主机才能改变 SDA 电平；SCL 高电平时，SDA 必须保持稳定**

- **SCL 低电平期间，主机可以安全改变 SDA 电平**  
- **主机每发送一个BIT位从机都会接收一个BIT**
- **每发送一个字节就需要从机进行应答否则立即停止传输，一个字节8个周期，那么第9个周期就是应答位** 



**数据发送逻辑**(依靠时钟线作为节拍器)**：**

- **写模式主机逻辑**：这里以起始信号(SCL高电平的同时拉低SDA)成功后为开始状态，此时主机要发送地址字节，一个字节有8个BIT位，首先发送第一个BIT位，先拉低时钟线(SCL)，然后把当前单比特电平设置到 SDA（一次只发 1 位，不是整字节），如果想要写入0，那么SDA就是低电平，如果想要写入1，那么SDA就是高电平，然后再拉高SCL( **这里是协议要求必须是在SCL低电平期间主机才能安全改变SDA电平**)，中间的间隔就是稳定数据，此时SCL为高电平，数据已经稳定下来，从机开始读取这个BIT数据，这里读取的时间是一个固定的时间，从机需要满足时序要求，在规定的时间期间读完数据，然后SCL拉低，主机开始写入数据位到SDA上面，写完后将SCL拉高，从机可以开始读取数据了，8 位字节发送完毕，第 9 个时钟：主机释放 SDA，总线被上拉为高电平，等待从机拉低 SDA 发出 ACK 应答，无拉低则为 NACK。
- **写逻辑细拆分：**
  - **起始状态：**以**起始信号**（SCL 高电平的同时，主机拉低 SDA）成功后，正式开始通信。
  - **核心前提**：主机需发送地址字节（1 个字节 = 8 个 BIT 位），单次仅发送 1 位，而非整字节。
  - **单 BIT 传输流程**（循环 8 次，完成 1 个字节发送）：
    - 第一步：主机先拉低时钟线（SCL）；
    - 第二步：主机将当前单比特电平设置到 SDA（发 0 则 SDA 为低电平，发 1 则 SDA 为高电平）；
    - 第三步：主机拉高 SCL（⚠️ 协议要求：必须在 SCL 低电平期间，主机才能安全改变 SDA 电平）；
    - 第四步：SCL 高电平期间，数据稳定，从机在规定时序内（固定读取时间）完成该 BIT 数据的读取；
    - 第五步：读取完成后，主机再次拉低 SCL，重复上述步骤，写入下一个数据位。
  - **应答判定**（8 位字节发送完毕后，第 9 个时钟）：
    - 主机释放 SDA 总线，总线通过上拉电阻拉至高电平；
    - 从机拉低 SDA → 发出 ACK 应答，可准备后续数据传输；
    - 从机未拉低 SDA（SDA 保持高电平） → 为 NACK 非应答。
- **读模式主机逻辑(从机输出数据，然后主机读取)**：SDA 只能在 SCL 低电平期间变化，因此发送方会在 SCL 低电平期间更新 SDA。(依据协议规定)，从机想要发送1就设备释放 SDA，由上拉电阻拉高，想要发送0就设备主动拉低 SDA，SCL 拉高后，数据进入稳定期，接收方需要在协议规定的建立/保持时间内完成数据采样。，最后接收到一字节后，从机会释放对SDA总线的控制，SDA会被上拉电阻拉至高电平，此时主机需要开始回应了，主机会拉低SCL，如果将SDA拉高(维持高电平)，那么就是无需再接收后续字节回应NACK，如果拉低则说明还需要继续读取数据，那么从机会继续发送。读完停止后**必须由主机发送停止信号**结束通信

- **读逻辑细拆分：**

  - **通用协议规则**
    - 依据 IIC 时序规定：**仅 SCL 低电平阶段，才允许更改 SDA 电平**，数据发送方统一在此时间段更新 SDA 状态。

  - **单比特数据传输流程**

    - 主机主动拉低 SCL，进入可修改 SDA 阶段
    - 从机设置输出电平
      - 发送数据 1：从机释放 SDA 引脚，依靠板载上拉电阻将 SDA 拉高
      - 发送数据 0：从机主动直接拉低 SDA 电平
    - 主机拉高 SCL，数据进入稳定状态
    - 主机在协议规定的时序范围内，完成该位数据采样读取
    - 主机再次拉低 SCL，准备接收下一位数据，循环执行

  - **单字节传输完成后的应答阶段**

    - 主机完整接收 8 位数据组成一字节后，**从机释放 SDA 总线控制权**

    - SDA 自动被上拉电阻拉为高电平

    - 主机拉低 SCL，由主机执行应答动作

      - 主机主动拉低 SDA = 发出 ACK：表示继续接收后续字节，从机持续发送数据

      - 主机维持当前 SDA 高电平不变 = 发出 NACK：表示停止接收数据

  - **通信收尾**
    - 主机发出 NACK 终止读取后，**必须由主机发送停止信号**，正式结束本次 IIC 读通信。



**什么是IIC()**</u>  

是一种集成电路之间的串行通信总线，属于半双工、同步串行总线，用时钟同步收发数据。那么**整体通信逻辑**就是，主机发起通信，维持 SCL 高电平，主机拉低 SDA，这个时候从机也知道主机拉低了数据线，进行监听状态，主机继续发送从机地址和读写位(7位的从机地址+1位的读写位)整合的一个8位字节，地址字节最低位为 0 = 主机向从机写数据；最低位为 1 = 主机从从机读数据，所有从机都会接收到这个字节并进行匹配，匹配成功的从机在确认自己被选中后准备后续交互，此时主机会释放SDA总线，总线通过上拉电阻拉至高电平，**此时主机会采样SDA的电平，那么现在会分为两种情况，第一种：**从机拉低了SDA数据线，那么说明从机发送了ACK应答信号，那么可以开始准备后续的数据传输了，**第二种情况：**没有从机拉低SDA，SDA依旧是高电平，说明没有从机回应，则为NACK 非应答，意味着可能地址错误，设备未上电，硬件故障，或者设备忙于通信，本次通信直接终止结束，等待下一次的完整通信。现在我们以从机拉低到低电平状态继续说，在应答成功的基础上依旧是分为两种情况：第一种：写模式，主机开始连续发送需要写入从机的业务数据，每发送完成一个完整 8 位字节，就等待一次从机 ACK，收到应答即可继续发送下一字节，无应答立刻终止传输。所有需要写入的数据全部发送完成后，主机结束数据发送流程。最后主机产生停止信号：保持 SCL 为高电平，将 SDA 从低电平拉高，宣告本轮 IIC 写通信正式结束，SCL、SDA 回归空闲高电平状态。**第二种：读模式**，（**这里的有个核心要点，首先需要主机先将寄存器地址写入到从机，然后再读取**）主机发送完设备寻址地址并收到 ACK 后，先发送要读取的从机寄存器地址，发送完成等待从机应答确认。从机回应ACK后，此时是写状态，所以需要切换到读状态，那么主机会发送重复起始信号，在不释放总线的情况选下(不释放总线，通信就不会被其他主机抢走)进行第二轮寻址，从机会重新进行匹配，并且进行回应ACK，总线数据控制权切换，**由从机主动控制 SDA 数据线**，从机按照主机提供的 SCL 时钟节拍，逐位向外输出内部存储的数据字节，从机每发送完一个 8 位完整数据字节后，交由主机回应答，如果主机主动拉低 SDA则 发出**ACK**，代表主机还要继续读取下一个数据，从机持续发送数据，如果主机保持 SDA 高电平则发出**NACK**，代表主机数据读取完毕，通知从机停止发送数据。主机发送 NACK 结束读取后，主机立即生成停止信号，拉高 SDA 结束整轮读通信，总线恢复空闲监听状态。



<u>**IIC死锁不停机解决方法：**</u>（一般出现在读模式下）

​	一般出现死锁场景为：DMA异常、MCU异常复位、中断嵌套、传感器跑飞

**通信前死锁检测方法：**

	1. 如果调用的标准库，通常内部会维护一个状态标志位用来进行判断，比如说STM32的IIC就会有一个 BUSY 位，我们可以通过开始通信的时候检测BUSY位，如果超时就代表出现了死锁
	1. 直接检测通信时间，时间超时就代表进入了死锁
	1. 直接读取SDA的电平来进行判断

**解决方法：**

 1. 关闭硬件IIC或者你写的软件IIC，防止干扰。 

    ```c
    HAL_I2C_DeInit();
    ```

 2. 手动使用GPIO 模拟时钟恢复总线，手动发送9个SCL脉冲，我们在不知道从机处于输出或者输入的哪个阶段，直接发送完整的脉冲，让从机能够跑完，完成剩余bit，到达ACK阶段，已经完成了一个字节的完整传输，此时为byte边界，从机开始交出SDA的总线控制权，然后主机发送停止信号，终止这一次的通信，此时通信恢复

    - 你手动发送的SCL脉冲一般会空出来几个，这几个就是空脉冲，从机已经释放SDA总线，SDA回到高电平，**且主机没有应答（重点）** ，此时从机就在等待主机应答，你发再多的SCL脉冲也只是空脉冲，不影响。







### 面试常见题目

1. **什么是 IIC？属于什么类型总线？同步还是异步？**

**答案**：IIC 是集成电路间串行通信总线，属于**半双工、同步串行总线**，依靠时钟线同步收发数据。



2. **IIC 一共几根线？作用是什么？**

**答案**：两根信号线 + 地线

- SCL：串行时钟线，提供通信时钟

- SDA：串行数据线，传输双向数据

- GND：共地

  

3. **IIC 为什么必须接上拉电阻？阻值一般多少？**

**答案**：IIC 引脚为**开漏输出**，只能拉低不能主动拉高，必须外接上拉电阻才能输出高电平；

常用阻值**4.7KΩ、10KΩ**。



4. **IIC 空闲状态电平？**

**答案**：空闲时**SCL、SDA 全部为高电平**。



5. **IIC 三种通信速率**

**答案**

- 标准模式：100Kbps

- 快速模式：400Kbps（最常用）

- 高速模式：3.4Mbps

  

6. **IIC 支持主从架构？**

**答案**：支持**一主多从、多主多从**，多主机具备总线仲裁功能。



7. **IIC 从机地址分为几种？**

**答案**：常用**7 位地址**，少数设备用 10 位扩展地址；7 位地址**左移一位**变成 8 位通信地址，最低位区分读写。



8. **读写地址规则**

**答案**：

- 7 位地址左移 1 位

- 最低位 0：**写地址**

- 最低位 1：读地址

  例：7 位 0x48 → 写 0x90 读 0x91

  

9.**简述起始信号**

**答案**：SCL 为高电平时，**SDA 从高电平跳变到低电平**，代表通信开始。



11. **数据传输规则**

**答案**：**高位优先发送**，SCL 高电平期间读取 SDA 稳定数据。



12. **应答信号 ACK/NACK**

**答案**

- 发送完 8 位数据后，主机释放 SDA，等待从机拉低 SDA 产生**ACK 应答**

- 无拉低即为**NACK 无应答**

- 应答位由**从机发出**

  

13. **什么是重复起始信号？作用**

**答案**：不发送停止信号，直接再次发送起始信号；

作用：**连续切换读写方向**，不用释放总线。



14. **软件模拟 IIC 原理**

**答案**：用普通 GPIO 口模拟 SCL、SDA 高低电平，通过延时严格模拟起始、停止、收发、应答全部时序，不使用片上外设。



15. **软件 IIC 优缺点**

**答案** 

优点：简单通用、任意 GPIO 可用、移植性强、几乎不出时序错误

缺点：占用 CPU 资源、速度慢、效率低



16. **硬件 IIC 优缺点**

**答案**

优点：外设自动收发、不占用 CPU、速度快

缺点：容易卡死总线、时序配置麻烦、兼容性差、易出现死锁



17. **硬件 IIC 最常见故障**

**答案**：总线死锁，SDA 被一直拉低，后续无法通信。(常见场景：MCU突然复位，程序崩溃，中断异常)
**解决方法：** 



18. **硬件 IIC 死锁复位方法**

**答案**：切换为 GPIO 模式，手动拉高 SDA/SCL，释放总线后重新初始化硬件 IIC。



19. **上拉电阻过大 / 过小问题**

**答案**

- 阻值太小：灌电流过大，烧坏引脚

- 阻值太大：上升沿太慢，高速通信出错

  

20. **3.3V 与 5V 设备 IIC 通信怎么兼容？**

**答案**：串联**分压电阻**，或使用**电平转换芯片**，禁止直接直连。



21. **A0/A1/A2 地址引脚作用**

**答案**：通过接高低电平修改设备低位地址，同一总线挂载多个同款 IIC 设备，避免地址冲突。



22. **IIC 读取不到 ACK 无应答，排查思路**

**答案**

1. 检查电源、地线是否连通
2. 检查上拉电阻是否焊接
3. 核对从机地址是否写错（最常见）
4. 检查 SDA/SCL 是否接反
5. 示波器查看起始时序是否正确
6. 硬件 IIC 判定总线是否卡死



23. **IIC 读出数据全 0 或全 FF**

**答案**

1. 设备未正常初始化
2. 寄存器地址填写错误
3. 读写时序颠倒
4. 没有等待设备就绪就读取



24. **IIC 时而通时而不通**

**答案**

1. 时序延时不足
2. 总线接触不良
3. 上拉电阻阻值不合适
4. 总线挂载设备过多负载过大



25. **如何用示波器抓取分析 IIC**

**答案**

1. 探头接 SCL、SDA，共地
2. 下降沿触发，调节时基
3. 识别起始、地址、应答、数据
4. 直接开启示波器 IIC 串行解码，自动解析地址和数据



26.**IIC、SPI、UART 区别**

**答案**

- UART：异步串口，2 线，一对一通信
- IIC：同步半双工，2 线，多设备，速度中等
- SPI：同步全双工，4 线，速度最快，一对一居多



27. **IIC 缺点**

**答案**：传输距离短、抗干扰弱、速率偏低、容易总线死锁。



28. **IIC 时钟拉伸是什么**？

**答案**：从机拉低 SCL 时钟线，**拉长低电平时间**，通知主机等待从机处理完毕再继续传输。



29. **多主机 IIC 仲裁机制**

**答案**：多个主机同时发数据时，发送高电平却检测到总线为低电平，判定仲裁失败，主动退出总线。



30. **RTOS 中使用 IIC 注意事项**

**答案**：必须加**互斥锁**，防止多任务同时抢占 IIC 总线造成时序错乱。



31. **IIC 读取传感器标准流程**

**答案**

1. 发送起始信号
2. 发送设备写地址，等待应答
3. 发送要读取的寄存器地址，等待应答
4. 发送重复起始
5. 发送设备读地址，等待应答
6. 连续读取数据
7. 主机发送非应答，发送停止信号



32. **IIC 扫描设备原理**

**答案**：遍历所有 7 位 IIC 地址，依次发送写地址，有 ACK 应答代表该地址存在设备。





## CAN通信原理

**两根线CANH和CANL的作用是什么？**

首先CAN使用的是差分通信(抗干扰能力强的原因之一)而不是单线电压，通过**两根线的电压差**来表示数据，默认状态无设备收发数据的时候会使（CAN_H ≈ 2.5V     |     CAN_L ≈ 2.5V），此时两根线没有电压差，这时候总线状态就是 **隐形位(逻辑1)** ，如果想要发送**显性位（逻辑0）** 就会调整电压差，CAN_H 上升，CAN_L 下降（CAN_H ≈ 3.5V | CAN_L ≈ 1.5V），此时电压差约等于2V，这种明显的电压差就代表**显性位（逻辑0）** 



**掩码的作用：**

是作为设备接收信息的摘选方式，它决定哪些 ID 的报文我要收进缓冲区给 CPU 处理，哪些报文直接丢弃忽略，通过**掩码（Mask）和过滤码（Filter）**的配合，对报文的 ID 进行“按位与”运算匹配



**核心仲裁机制：** 

- 设备使用CAN通信会优先发送ID到总线上判断不同设备发送的优先级，设备是便发送边检测，总线上面的显性位时0，隐形位是1，所以总线会优先显示显性位，如果设备发送的是1，但是总线显示的是0，说明有其他设备在发送，且优先级比你高，那么这个设备就会退出竞争，所以不同设备同时发送信息到CAN总线上，我们需要仲裁谁先来发送，比如A设备的ID为：0x100（二进制：0001 0000 0000），B设备的ID为：0x200（二进制：0010 0000 0000），**从高位开始判断**，前面两个BIT位都是0，而到第三位的时候A为0，而B为1，此时0的优先级更高，所以A设备的优先级更高开始发送数据，B设备退出竞争并转为接收模式等待总线变为空闲状态，再次发起传输，并且从这里可以推断出 **ID 越小 优先级越高** 。

  -  但是这里有个核心要点，ID和数据是一起传输的，而不是先发送ID抢占位置再发送数据，ID比对，谁优先级高谁接着发

  

**一个CAN通信标准帧结构：**

| 组成部分        | 长度 (位) | 核心作用与内容                                               |
| --------------- | --------- | ------------------------------------------------------------ |
| 1. 帧起始 (SOF) | 1 bit     | 标志开始。显性电平（0），告诉总线上的其他人“我要开始说话了”。 |
| 2. 仲裁段       | 12 bits   | 决定优先级。包含 11位标识符 (ID) + RTR位 (区分是数据还是请求)。ID 越小，优先级越高。 |
| 3. 控制段       | 6 bits    | 定义数据长度。包含 IDE 位、保留位 r0 和 DLC (数据长度代码，0-8)。 |
| 4. 数据段       | 0~64 bits | 有效载荷。实际要传输的数据，长度为 0 到 8 个字节。           |
| 5. CRC 段       | 15+1 bits | 校验完整性。15位 CRC 校验码 + 1位 CRC 界定符，用于检测传输错误。 |
| 6. 应答段 (ACK) | 2 bits    | 确认接收。发送方发隐性（1），接收方拉低（0）表示“我收到了”。 |
| 7. 帧结束       | 7 bits    | 结束标志。全是隐性电平（1），表示报文结束。                  |



**时序配置：同步机制**

- **引入名词：**
  - **时间量子（`Time Quantum，TQ`）**：CAN里的最小时钟刻度，时间单位，可以用来描述一个BIT所占用的时间
    - 预分频器（`Prescaler`）：用来计算TQ的具体时间，假设CAN的外设时钟是36MHZ，`Prescaler` 给到9，那么TQ = 9 / 36MHz  = 250ns，一个TQ就是250ns
  - **同步段 Sync_Seg** ： 固定1TQ的时间，当A设备想要发送消息的第一个BIT位是SOF帧，各节点在检测到 SOF 边沿后，以该边沿作为本地 bit 时间同步参考，然后开始按自己本地时钟跑 ：   **1TQ(Sync) + BS1 + BS2** 
  - **传播段：** 通常与相位段一起赋值
  -  **相位段 Phase_Seg1：** 采样前缓冲区，可动态调整，缓冲区结束后进行采样
  -  **相位段 Phase_Seg2：**采样后缓冲区，等待进入下一bit
  -  **SJW（重同步跳转宽度）**：动态调整的时候一次允许调整多少TQ，如果当前发现边沿比预计晚2TQ，那么CAN 可能拉长当前bit 2TQ，但是不同无限调整，只是为了在产生微小误差的时候能够快速调整回来
  - **位填充：**  在仲裁 / 控制 / 数据 / CRC 帧做填充，SOF/EOF/ACK 帧不填充，发送端硬件会扫描原始比特流，遇到连续 5 个相同位（0/1） 后会自动插入 1 个相反位（填充位），接收端（硬件）收到比特流后，先扫描：遇到连续 5 个相同位 → 后面那个一定是填充位，直接删掉；
- **重同步机制：**
  - **CAN 没有独立时钟线，因此每个节点都依靠本地时钟估算 bit 时间，但不同节点时钟一定会存在微小误差，所以 CAN 会利用总线边沿不断进行重同步，内部有位填充机制保障了跳变沿必定会产生。 **  
  - **如果实际边沿到来的时间比本地预估时间更早，说明本地时钟偏慢，此时 CAN 会缩短当前 bit 时间（通常缩短 Phase_Seg1），让采样点提前，重新跟上总线节奏。**   
  - **如果实际边沿到来的时间比本地预估时间更晚，说明本地时钟偏快，此时 CAN 会拉长当前 bit 时间（通常拉长 Phase_Seg2），让当前 bit 结束得更晚，从而重新同步总线时序。 ** 
    - **如果实际边沿跳变来的时间比你的预估时间晚，但是仍然位于当前的BIT位区域，只是和你设置的采样点有一点点误差，这样慢慢积累下去会到达错误的BIT位边界，所以需要动态调整，并且这也是工程上要求采样点在 75%-85% 之间的原因**
  - **SJW（重同步跳转宽度）用于限制一次重同步最多允许调整多少个 TQ，本质上是限制 bit 时间最大修正幅度，避免同步过程震荡过大。 ** 
  - **由于正常晶振误差通常很小，并且 CAN 会频繁利用边沿进行同步，因此一般情况下 SJW 设置为 1TQ 就已经足够。**   

- **时间计算：**

  - **CAN 协议规定，每一位的时间（Bit Time）必须分成 4 个阶段，顺序固定：**

    - ```
      位时间 = Sync_Seg（1TQ） + Prop_Seg（传播段） + Phase_Seg1（相位缓冲段1） + Phase_Seg2（相位缓冲段2）
            = Sync_Seg（1TQ） + Time Quanta in Bit Segment 1 + Time Quanta in Bit Segment 2
      ```

    - 传播段 Prop_Seg  + 相位段 Phase_Seg1  = 位段1 时间量子(`Time Quanta in Bit Segment 1`)     以TQ为单位
    -  相位段 Phase_Seg2  =  位段2 时间量子(`Time Quanta in Bit Segment 2`) 	            	         以TQ为单位
    - TQ具体时间计算： `Prescaler / CAN的外设时钟频率` 

  - 比特率计算：

    - ```c
      比特率 = 1/位时间   // 比特率计算
      ```

  - 采样点位置计算：

    - ```c
      采样点位置 = (Sync_seg + BS1) / 位时间  = (1 + BS1) / (1 + BS1 + BS2)
      ```

    - 如果采样点过早（BS1 太短）或过晚（BS2 太短），节点可能无法正确读取总线上的位值，导致错误帧或通信失败，造成卡死的情况，**所以工程上要求采样点在 75%-85%，以平衡传播延迟和相位误差。(实测STM32F103在1MHZ的情况下，BS2的TQ不要低于2）** 



**常见异常管理:**  

- ACK错误 → 判断节点是否在线
- Error Warning/Passive → 判断总线是否开始恶化
- Bus-Off →  通过错误数量判断节点是否已经失控
- FIFO溢出 → 判断CPU是否跟不上总线
- Timeout → 判断通信链路是否阻塞



**四种工作模式：** 

- 正常模式： 可向总线发送数据或者从总线接收数据
- 回环模式： 发送的数据直接到输入(总线来检测数据)，不同从总线接收数据
- 静默模式： 只可向总线发送数据1，不可发送数据0，可以从总线接收数据
- 回环静默模式： 发送数据到输入，总线不能监测到数据(只有自身设备可以检测到，相当于私密测试)，也不能从总线接收数据





**CAN控制器基本参数解析：**  

1. **DBF（Debug Freeze）调试冻结 **                                                    **重点**

   - 用于调试阶段。

   - 当 MCU 进入 Debug 断点暂停时，是否同时冻结 CAN 外设。

   - 如果不开启，CPU停下后 CAN 仍然继续运行，可能导致：
     - FIFO溢出
     - ACK失败
     - Error Frame
     - Bus-Off

   - 工程上调试阶段一般开启，避免断点期间总线异常。

     

2. **TTCM（Time Triggered Communication Mode）时间触发模式**    **重点**

- 给 CAN 报文增加时间戳，实现时间同步通信。
- CAN 控制器内部会维护时间计数器，并在发送/接收帧时记录时间。
- 常用于：
  - 汽车实时控制
  - 工业同步系统
  - ISO11898-4 时间触发CAN
- 普通 STM32 项目很少使用，一般关闭。

------

3. **ABOM（Automatic Bus-Off Management）自动离线恢复**

- 当节点错误过多时，CAN 会进入 Bus-Off 状态。
- Bus-Off 后节点停止发送和接收，避免继续污染总线。
- ABOM 用于控制：
  - 是否允许 CAN 自动从 Bus-Off 恢复。

- **工程理解：**
  - 开启：
    - 节点故障后自动恢复，如果是瞬态干扰，线闪了一下，那么自动恢复后他会继续正常工作，如果硬件死机，就记录日志，连续超过三次直接停机，软件无法修复硬件故障只能避免瞬态电磁干扰
    - 适合普通设备、小型项目

------

4. **AWUM（Automatic Wake-Up Mode）自动唤醒**       		**重点**

- CAN 支持低功耗睡眠模式。
- 开启后：
  - 当总线检测到通信活动时自动唤醒。
- 常用于：
  - 汽车ECU
  - 电池设备
  - 低功耗节点

工程意义：

节点休眠后无需软件轮询检测，可由总线活动自动唤醒。

------

5. **NART（No Automatic Retransmission）自动重发控制**

- CAN 默认机制

- 发送失败后：CAN 会自动重发。

  - ACK失败

  - 仲裁失败

  - 位错误

- NART作用： 是否禁止自动重发。

  - NART = ENABLE
    - 禁止自动重发

  - NART = DISABLE
    - 允许自动重发（常用）

- **工程理解：**

  - 普通控制系统：

    - 一般允许自动重发：

    - 提高可靠性。

  - 强实时系统：

    - 有些数据过期即无意义，例如：电机瞬时控制量，高频状态量

    - 可以关闭自动重发，避免旧数据堆积。



------

6. **RFLM（Receive FIFO Locked Mode）FIFO锁定模式**

- 用于解决接收FIFO满了的问题

- **两种策略**： 

  - **锁定模式 Enable**

    - FIFO满后：

      - 保留旧数据

      - 丢弃新数据

  - 适用于：流程事务数据，比如说参数配置或者Bootloader升级
    - 我们对于这种流程事务数据一般会配有状态机，但是如果短时间FIFO爆了，新的数据覆盖了旧的数据，顺序乱了，状态机也很难处理，但是如果顺序没有乱，状态机就可以正常检测后面缺了哪个数据重新发即可

  - **非锁定模式 Disable**
    - FIFO满后：
      - 新数据覆盖旧数据
  - 适用于：IMU、速度、电流、状态量，这些数据新数据更重要

------

7. **TXFP（Transmit FIFO Priority）发送优先级方式**

- 用于控制：多个发送邮箱同时等待发送时，谁优先。

- Disable（默认）

  - 按 CAN ID 优先级发送：

    - ID越小

    - 优先级越高

  - 符合 CAN 仲裁机制。

- Enable

  - 按进入邮箱顺序发送：

    - 谁先进

    - 谁先发

- **工程上一般：Disable，保持 CAN 原生优先级机制。**  





**工程核心理解问题描述：**

- 总线异常怎么办
  - CAN 总线出现异常时，硬件控制器会先自动进行错误检测与恢复，例如 Error Frame、自动重发、错误计数、Error Passive、Bus-Off 等机制，因此软件层真正需要做的是监控总线状态并决定系统如何恢复。工程中通常会通过错误中断统计 ACK错误、CRC错误、Stuff错误、Bus-Off 等异常类型，并根据错误严重程度执行对应策略，例如降低发送频率、重启 CAN 控制器、切换系统到安全模式、停止电机输出等。本质上，CAN硬件负责发现错误，而软件负责保证系统在异常情况下依旧能够稳定运行。

- FIFO满了怎么办
  - FIFO 满说明 MCU 处理报文的速度已经跟不上总线到达速度，此时必须决定优先保留什么类型的数据。如果是 IMU、电流、速度等状态流数据，通常保留最新数据更合理，因为控制系统只关心当前状态；如果是 Bootloader、配置命令、多包协议等事务流数据，则更倾向于保留旧数据，因为旧数据决定当前协议上下文和流程状态。真正工程中不会单纯依赖 FIFO 策略解决问题，而是会通过优化中断优先级、减少无效报文、合理配置过滤器、降低总线负载等方式避免 FIFO 溢出。本质上 FIFO 策略是在系统已经处理不过来的情况下，决定优先保证实时性还是流程完整性。
- 节点掉线怎么办
  - CAN 是广播总线，没有 TCP 那种连接状态，因此 CAN 控制器本身并不知道哪个节点在线或离线。工程中通常通过 Heartbeat（心跳包）机制维护节点在线状态，即每个节点周期性发送“我还活着”的状态帧，如果主控在规定时间内未收到某节点心跳，就认为该节点掉线。节点掉线后系统通常会执行安全策略，例如停止运动、关闭输出、进入降级模式或报警提示。本质上，节点在线管理属于软件层逻辑，而不是 CAN 协议本身负责的内容。
- Bus-Off怎么办
  - 当节点持续产生错误时，CAN 会认为该节点已经开始污染总线，因此当发送错误计数 TEC 超过阈值后，控制器会进入 Bus-Off 状态，此时节点停止发送、停止 ACK、退出总线。工程中软件需要检测 Bus-Off 状态并决定恢复策略，例如重新初始化 CAN、限制恢复次数、防止节点反复上线污染总线，严重情况下甚至需要彻底关闭该节点并等待人工处理。Bus-Off 本质上是 CAN 协议为了保护整个总线而主动隔离故障节点的一种自保护机制。
- 发送失败怎么办
  - CAN 的“发送成功”并不代表业务一定成功，它只能说明报文被控制器尝试发送到总线上。发送失败可能来自 ACK失败、仲裁失败、总线繁忙、波特率不一致等原因。CAN 控制器默认会自动重发，但工程中软件仍然需要增加发送超时机制、防止发送邮箱长期卡死，同时很多高可靠系统还会在应用层增加 ACK 应答、超时重发、序号校验等机制，确保业务逻辑真正执行成功。本质上，CAN 链路层只能保证“尽力发送”，真正的业务可靠性需要应用层协议保证。
- 低功耗怎么办
  - 很多 CAN 节点需要长期在线运行，例如汽车 ECU、电池管理系统、工业传感器，因此低功耗管理非常重要。CAN 支持睡眠模式，节点进入休眠后可显著降低功耗，而 AWUM（自动唤醒）允许节点在检测到总线活动后自动恢复工作。工程中常见场景是汽车熄火后 ECU 进入休眠，但检测到钥匙、车门或启动信号后自动唤醒。本质上，CAN 不仅仅是通信协议，还需要考虑节点长期运行过程中的功耗管理与系统唤醒机制。
